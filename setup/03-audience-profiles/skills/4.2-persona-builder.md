# Skill 4.2 — Persona Builder (setup-audience-persona-builder)

## Papel

Cria personas detalhadas para cada segmento de público identificado na Skill 4.1. Opera em modo de **AUTO-GERAÇÃO**: lê os arquivos de contexto, produz um rascunho completo com ~80% dos campos preenchidos por inferência e pede validação em blocos — o usuário corrige e complementa, não cria do zero. Integra pesquisas R2 (demográfica), R3 (comportamento de busca) e R6 (inferência psicográfica).

**Output que preenche:** Seção 2 (Persona) de cada `AUDIENCE_[nome].md`, incluindo: perfil base, contexto de vida/trabalho, dados socioeconômicos, 5 Rings of Buying Insight, perfil psicográfico, linguagem e comunicação.

---

## Quando Acionar

- "criar persona"
- "construir persona para [público]"
- "detalhar o público [nome]"
- Chamada internamente após Skill 4.1 pelo orquestrador

---

## Inputs

**Obrigatórios:**
- Output da Skill 4.1 (sumário de públicos do `WORKFLOW_STATE.md`)
- `PRODUCT_CONTEXT.md` — para inferir cargo, renda, dores por pricing
- `BRAND_VOICE.md` — para calibrar como a persona fala e o que ressoa

**Opcionais:**
- `COMPANY_PROFILE.md` — stage da empresa afeta complexidade das personas

**Não carregar:**
- `AUDIENCE_*.md` de outros públicos — evitar contaminação entre personas

---

## Procedimento

**Processar um público-alvo por vez.**

### Passo 1 — Leitura de Contexto e WORKFLOW_STATE
Ler `PRODUCT_CONTEXT.md`, `BRAND_VOICE.md` e o `WORKFLOW_STATE.md` antes de qualquer pergunta.

Do WORKFLOW_STATE.md, identificar:
- Qual segmento está sendo processado agora (`publico_ativo`)
- Se é B2B com bifurcação (`b2b_bifurcacao: true`) — ajustar rascunho para decisor ou usuário conforme `perfil_ativo`
- Status das pesquisas anteriores (`r1_status`, `r1_lista_concorrentes`) — usar dados da R1 para enriquecer o rascunho

### Passo 2 — Geração do Rascunho Completo
**ANTES de qualquer pergunta**, gerar rascunho completo da persona por inferência. Marcar todos os campos com `[E]` (Estimativa). Usar `references/generations-guide.md` para pré-preencher automaticamente campos de geração, hobbies, referências culturais e redes sociais assim que faixa etária for inferida.

**Regras de inferência campo a campo:**

| Campo | Como inferir |
|-------|-------------|
| Nome arquétipo | Combinar cargo + traço emocional dominante. Ex: "Carla, a gestora sobrecarregada" |
| Faixa etária | Pricing alto (>R$500/mês) → 28-45 anos; produto operacional simples → mais amplo |
| Geração | Derivar da faixa etária → consultar `references/generations-guide.md` automaticamente |
| Cargo | Problema central do produto → cargo que sente essa dor mais intensamente |
| Tamanho da empresa | Pricing + modelo de negócio → PME (R$100-500/mês), Mid-market (>R$500/mês), Enterprise (>R$2k/mês) |
| Localização | B2B → capitais e centros econômicos; B2C → distribuição nacional salvo nicho regional |
| Escolaridade | Cargo de gestão → superior completo; técnico/operacional → técnico ou superior |
| Renda individual | Pricing do produto / 0,15 (produto não deve ultrapassar 15% da renda disponível para compra planejada) |
| Classe social | Derivar da renda estimada: A (>R$20k/mês), B1 (R$10-20k), B2 (R$5-10k), C1 (R$2,5-5k) |
| Valores | BRAND_VOICE arquétipo → selecionar 3 valores alinhados. Ex: Arquétipo Sábio → precisão, aprendizado, autonomia |
| Relação com mudança | Produto SaaS inovador → early adopter. Produto de mercado estabelecido → early majority |
| Tom de comunicação | BRAND_VOICE.md arquétipo → tom compatível |
| Canais | Faixa etária + cargo → consultar mapa de plataformas em `references/persona-questions.md` (P8.6) |
| Hobbies / Referências culturais | Geração + classe social → consultar `references/generations-guide.md` |
| P8.1 — Palavras que usa | Problema central reescrito na voz do sintoma (não da categoria). Ex: "marketing que não rende" em vez de "baixo ROI" |
| P8.2 — Palavras que afasta | Jargões corporativos, termos de vendedor forçado. Ex: "solução disruptiva", "transformação digital" |
| 5 Rings (rascunho inicial) | Ring 1: pressão externa recente relacionada ao problema. Ring 2: métrica que valida sucesso para este cargo. Ring 3: risco de errar a escolha / custo de mudar. Ring 4: pesquisa via Google + pares + LinkedIn. Ring 5: prova do mesmo perfil + garantia |

**Sugestões automáticas adicionais:**
- B2B enterprise com decisor separado → persona tem influência mas não decisão unilateral — Ring 3 inclui "aprovação interna"
- Produto de produtividade → geração Millennial → referências culturais anos 90/00

### Passo 3 — Verificação de Dados Internos
Antes de executar pesquisas externas, perguntar:
> "Você tem Google Analytics, Instagram Insights ou dados do CRM com informações sobre sua audiência? Me passe — são mais precisos que qualquer pesquisa externa."

- Se forneceu dados internos → processar como `[U]` na seção R2

### Passo 4 — Execução das Pesquisas
Após Bloco A (Demographics) validado pelo usuário — faixa etária e região confirmadas:
- Executar **R2** (demográfica) + **R3** (comportamento de busca) **em paralelo**
- Após R2 confirmada → executar **R6** (inferência psicográfica — SEM web search)
- Gravar R2, R3, R6 no arquivo `[ACTIVE_COMPANY_PATH]/audiences/RESEARCH_[slug-publico].md`
  - Se o arquivo não existir: criar usando `references/research-template.md`
  - Adicionar seções `## R2`, `## R3`, `## R6` ao arquivo existente
- Atualizar rascunho da persona com os dados confirmados (marcação [D] ou [U])

### Passo 5 — Validação em 3 Blocos
Apresentar o rascunho em 3 blocos para validação (não todos de uma vez):

**Bloco A — Demographics:**
> "Aqui está meu rascunho do perfil demográfico de [Nome Arquétipo]. O que está certo, o que muda?"
[Apresentar: faixa etária, geração, cargo, empresa, localização, escolaridade, renda, classe social]
*→ Após validação do Bloco A: executar R2 + R3 em paralelo (Passo 4)*

**Bloco B — Psychographics:**
> "Agora o perfil psicográfico e de valores. Ajuste o que não se encaixa."
[Apresentar: valores, relação com mudança, relação com autoridade/risco, vieses, referências que admira, hobbies, referências culturais]
*→ Após validação do Bloco B: executar R6 (Passo 4)*

**Bloco C — Communication:**
> "Por último, como essa persona se comunica e consome conteúdo."
[Apresentar: P8.1 — palavras que ela usa para descrever o problema, P8.2 — palavras que a afastam, P8.3 — tom esperado, P8.4 — canais não-sociais, P8.5 — formatos preferidos, P8.6 — redes sociais com intenção de uso]

### Passo 6 — Perguntas Complementares
Perguntar APENAS os campos que a skill não consegue inferir — **uma por vez**, aguardar resposta:

**P2.6 —** "Esse público costuma pesquisar seu tipo de problema fora do horário comercial (noite/madrugada)? O que buscam?"
*(⚠️ Resposta reservada para Skill 4.3 — camada de dores)*

**P2.7 —** "Existe alguma vergonha ou tabu ligado ao problema que o produto resolve?"
*(⚠️ Resposta reservada para Skill 4.3 — Camada 3)*

**P2.8 —** "Se essa persona fosse a um jantar com profissionais muito bem-sucedidos do mesmo setor, o que ela diria que faz? Como gostaria de ser vista?"
*(⚠️ Resposta reservada para Skill 4.3 — Desejo de Identidade)*

**P3.2 —** "Como essa persona reage a mudanças? Adota novidades cedo ou espera provas?"

**P3.9 —** "Qual frase ela diz internamente sobre por que ainda não resolveu o problema?"
*(⚠️ Resposta reservada para Skill 4.3 — Narrativa Interna)*

**P7.1 —** "O que aconteceu que fez alguém como ela buscar solução agora — não 'algum dia', mas agora? Qual foi o evento gatilho?"
*(Alimenta Ring 1 dos 5 Rings)*

**P7.2 —** "Como ela pesquisa antes de comprar? Google, pede indicação para colegas, grupo de WhatsApp, LinkedIn?"
*(Alimenta Ring 4 dos 5 Rings)*

**P7.3 —** "Antes de comprar, quem mais ela consulta? Quem tem poder de veto ou de influência na decisão?"
*(Alimenta Ring 4 — especialmente relevante em B2B)*

### Passo 7 — 5 Rings of Buying Insight (Adele Revella)
Aplicar os 5 Rings a cada persona, usando as respostas dos Passos 5 e 6 como base:

- **Ring 1 — Gatilho** ← P7.1: "O que fez buscar solução AGORA? (evento específico, não genérico)"
- **Ring 2 — Sucesso** ← P5 + cargo: "O que significa sucesso para ela? Qual métrica valida que funcionou?"
- **Ring 3 — Barreira** ← P3.11 + P7.4 (coletado em 4.3): "O que pode impedir a compra? Objeção real vs. falada?"
- **Ring 4 — Jornada** ← P7.2 + P7.3: "Como pesquisou e avaliou? Quem envolveu no processo?"
- **Ring 5 — Decisão** ← P7.5 + P3.12: "O que seria decisivo para escolher ESTE produto especificamente?"

*Rings que dependem de dados coletados em 4.3 (P7.4, P7.5) → pré-preencher com `[E]` agora e atualizar em 4.3.*

### Passo 8 — Reserva Explícita para Skill 4.3
As respostas de P2.6, P2.7, P2.8 e P3.9 **não vão para a Seção 2 (Persona)**. Vão para a Seção 3 (Dores). Gravar no `WORKFLOW_STATE.md` com a estrutura exata:

```markdown
## Campos Reservados para Skill 4.3 — [Nome do Público]
- p2_6_pesquisas_noturnas: "[resposta literal]"
- p2_7_vergonha: "[resposta literal]"
- p2_8_jantar_bem_sucedidos: "[resposta literal]"
- p3_9_narrativa_interna: "[resposta literal]"
```

### Passo 9 — Geração do Output Final
1. Verificar se `[ACTIVE_COMPANY_PATH]/audiences/AUDIENCE_[slug-publico].md` existe:
   - Se não existe: criar com o cabeçalho do template `references/output-template.md` e preencher Seção 1 do WORKFLOW_STATE (copiando o sumário da Skill 4.1)
   - Se existe: expandir adicionando a Seção 2
2. Preencher Seção 2 completa usando o template `references/output-template.md`
3. Campos P2.6, P2.7, P2.8, P3.9 → gravar em WORKFLOW_STATE (Passo 8), não em AUDIENCE_[nome].md ainda

## Comportamentos Especiais

**MODO AUTO-GERAÇÃO OBRIGATÓRIO:** Sempre gerar rascunho ANTES de fazer perguntas. A persona nunca é criada do zero pelo usuário — é refinada a partir de um rascunho gerado pela IA.

**VALIDAÇÃO POR BLOCOS:** Nunca apresentar todos os campos de uma vez — dividir em 3 blocos (Demographics, Psychographics, Communication). Isso evita fadiga e mantém qualidade das respostas.

**DADOS INTERNOS PRIMEIRO:** Sempre perguntar sobre dados do usuário (Analytics, CRM) antes de executar qualquer pesquisa externa.

**MARCAÇÃO DE CONFIANÇA:** Usar o sistema de marcação:
- `[H]` — Hipótese estruturada (inferência do LLM)
- `[E]` — Estimativa (padrões, sem dados diretos)
- `[D]` — Dado de pesquisa (fontes verificadas)
- `[U]` — Input direto do usuário
- `[V]` — Validado (confirmado pelo usuário)

**CONTEXTUALIZAR RESPOSTAS:** Se a persona disser que faz X, perguntar "por que isso importa para ela?" para ir além do óbvio.

**ATUALIZAR WORKFLOW_STATE ao final com:**
- `persona_ativa`: nome arquétipo + segmento
- `audience_file`: path do AUDIENCE_[nome].md criado
- `campos_reservados_4_3`: estrutura do Passo 8 (p2_6, p2_7, p2_8, p3_9)
- `r2_status`, `r3_status`, `r6_status`: executada / pulada
- `rings_pendentes`: [3, 5] — rings que dependem de dados da Skill 4.3
- `proxima_acao`: "Skill 4.3 — Pain & Desire Mapper"
